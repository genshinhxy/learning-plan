# GraphRAG-Agent 主项目路线图

> **项目名**：面向复杂文档问答的 GraphRAG-Agent 系统
> **定位**：简历代表作 + 论文主战场。研究新意与工程闭环并重。
> **时长**：第 4-14 月（P1 v1 跑通 → P2 深化成文）

---

## 0. 为什么是这个项目

- 卡在你最稀缺的交叉点：知识图谱 + 大模型 + Agent 推理。
- 单纯 RAG 已经卷烂；把**图结构真正接进检索与推理链路**，才有研究空间和差异化。
- 一个项目同时喂养论文（创新点 + 消融）和就业（端到端系统 + 部署）。

---

## 第 1 步：选题与基线（第 4 月）

### 做什么
1. 跑通 **LightRAG** 官方 demo（首选脚手架，轻量、代码清晰）。
2. 跑通 **Microsoft GraphRAG** demo（理解工业级实现，对比设计差异）。
3. 选定数据域：建议从**学术文献问答**或**多文档技术手册问答**起步（你熟悉、容易构造评测）。
4. 建立 baseline：朴素向量 RAG 的问答效果作为对照组。

### 必读论文
- arXiv:2404.16130 Microsoft GraphRAG
- arXiv:2410.05779 LightRAG
- arXiv:2312.10997 RAG Survey（建立全景）
- arXiv:2501.00309 GraphRAG Survey

### 产出
跑通两个 GraphRAG baseline + 一个朴素 RAG 对照，记录初始指标（准确率、召回、延迟）。

---

## 第 2 步：知识图谱构建（第 5 月）

### 做什么
1. 文档切块 → 实体抽取（GLiNER）→ 关系抽取（REBEL/GLiREL）→ 三元组。
2. 三元组写入 Neo4j，设计 schema（实体类型、关系类型、属性）。
3. 处理实体消歧/对齐（同名实体合并），构建社区/层次结构（参考 GraphRAG 的 community detection）。
4. 关键决策：图的粒度（细到句子级 vs 粗到文档级）直接影响后续检索质量。

### 必读论文
- arXiv:2306.08302 KG+LLM Roadmap
- arXiv:2405.14831 HippoRAG（神经生物学启发的图记忆，多跳很强）
- arXiv:2401.18059 RAPTOR（层次化摘要树，可与图结合）

### 产出
一个从原始文档自动构建的领域知识图谱（Neo4j），可视化展示 + 质量抽检报告。

---

## 第 3 步：混合检索（第 6 月）

### 做什么
1. 三路召回：向量检索（BGE-M3）+ 图检索（Cypher 多跳/子图）+ BM25 关键词。
2. 融合策略：RRF（倒数排名融合）或加权融合。
3. 重排序：bge-reranker-v2-m3，或上 ColBERT 后期交互。
4. 对比实验：单路 vs 多路、有无 rerank 的指标差异。

### 必读论文
- arXiv:2004.12832 ColBERT / arXiv:2112.01488 ColBERTv2
- BGE/FlagEmbedding：https://github.com/FlagOpen/FlagEmbedding
- arXiv:2509.10697 结构化 RAG 综述

### 产出
混合检索模块，检索召回率显著优于单路基线，有量化对比表。

---

## 第 4 步：Agent 推理（第 7 月）

### 做什么
1. 用 **LangGraph** 编排 Agent 状态机。
2. 实现 ReAct 循环：思考 → 调用检索工具 → 观察 → 再思考。
3. 多跳推理：让 Agent 沿图结构走路径（而非一次性塞所有上下文）。
4. 加 Reflexion 式自我反思：检索不足时自动改写 query 重检索。
5. 工具集：图查询工具、向量检索工具、子图扩展工具。

### 必读论文
- arXiv:2210.03629 ReAct
- arXiv:2303.11366 Reflexion
- arXiv:2309.07864 Agent Survey
- arXiv:2302.04761 Toolformer

### 产出
能做多跳推理的 Agent，对复杂问题（需跨多个文档/实体）的回答优于单步 RAG。

---

## 第 5 步：评测体系（第 8 月）

### 做什么
1. 构造评测集：50-100 个问题，分单跳/多跳/需图推理三类，标注标准答案。
2. 硬指标：答案准确率、检索召回、引用正确性、延迟、token 成本。
3. 软指标：LLM-as-Judge 评答案质量（准确性/完整性/相关性）。
4. 全面对比：朴素 RAG vs GraphRAG baseline vs 你的系统。
5. 复用已有方法论：`资源/第三阶段-评估验证/AI工程评估方法论.md`。

### 必读 / 参考
- arXiv:2308.03688 AgentBench、arXiv:2311.12983 GAIA（评测设计参考）
- 已有评估方法论文档（LLM-as-Judge、回归测试、trace）

### 产出
完整评测报告 + 评测集（评测集本身可作为论文贡献之一）。

---

## 第 6 步：研究新意 + 消融（第 9-12 月）

### 创新点候选（择一深挖，这是论文的核心）
1. **图结构指导检索路径**：用图拓扑（中心性/路径权重）引导 Agent 选择检索方向，vs 盲目 BFS/DFS。
2. **KG 与文本检索的融合/冲突消解**：当图谱事实与文本检索结果矛盾时如何裁决。
3. **图拓扑做可信度评估**：用子图结构特征给检索结果打可信分，过滤幻觉。
4. **领域适配的 GraphRAG**：针对特定领域（学术/法律/医疗）的图构建与检索定制。

### 做什么
1. 选定 1 个创新点，形成清晰的假设（"加了 X，多跳问题准确率应提升"）。
2. 设计消融实验：逐个去掉模块，证明每个设计的贡献。
3. 在 2-3 个公开数据集 + 自建集上验证泛化性。
4. 误差分析：失败案例归因（哪类问题还不行、为什么）。

### 产出
有说服力的实验结果 + 消融表 + 误差分析，达到投稿标准。

---

## 第 7 步：投稿 + 工程化（第 13-14 月）

### 论文侧
1. 先投 workshop（快速反馈），再冲主会（ACL/EMNLP/NeurIPS/AAAI）。
2. 写作结构：动机 → 方法 → 实验 → 消融 → 误差分析。
3. 开源代码 + 评测集（增加影响力和可信度）。

### 工程侧（同步做，为求职）
1. FastAPI 包成服务，docker-compose 编排（API + Neo4j + 向量库 + vLLM）。
2. 加 trace（每次问答的检索路径、token、延迟可视化）。
3. 压测：并发下的延迟/吞吐，加缓存（Prompt 缓存、检索结果缓存）。
4. 写一份系统架构文档 + README，面试时能直接讲。

### 产出
- 一篇投稿
- 一个可演示、可部署的完整系统
- 简历上的"代表作 1"
