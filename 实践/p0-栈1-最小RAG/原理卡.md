## 1. 文本嵌入（Embedding）

### 解决什么问题
计算机不懂文字，只懂数字。嵌入把一段文本映射成一个固定维度的稠密向量，让"语义相近的文本 → 向量空间中距离相近"。这是检索的前提：把"找语义相关的文档"变成"找向量空间里离得近的点"。

你代码里的 `BAAI/bge-small-zh-v1.5` 就是个嵌入模型，输出 512 维向量。

### 核心原理（含数学）
嵌入模型本质是一个编码器 $f_\theta$，把文本 $t$ 映射到向量 $\mathbf{v} \in \mathbb{R}^d$：

$$\mathbf{v} = f_\theta(t), \quad \mathbf{v} \in \mathbb{R}^d$$

现代嵌入模型（bge、E5 等）是基于 BERT 类 Transformer 编码器，训练目标是**对比学习**（contrastive learning）。给一个查询 $q$、一个正样本 $d^+$（相关文档）、若干负样本 $d^-_i$（不相关），用 InfoNCE 损失：

$$\mathcal{L} = -\log \frac{\exp(\text{sim}(q, d^+)/\tau)}{\exp(\text{sim}(q, d^+)/\tau) + \sum_i \exp(\text{sim}(q, d^-_i)/\tau)}$$

其中：
- $\text{sim}(a,b)$ 是相似度（通常余弦相似度，见第 2 节）
- $\tau$ 是温度系数，控制分布平滑度。$\tau$ 越小，模型对"正负样本相似度差距"越敏感（拉得越开）。

**这个损失在干什么**：分子是正样本对的相似度，分母是正+所有负样本。最小化 $\mathcal{L}$ 等价于"让正样本相似度在所有候选里占的比例尽量大"——即把正样本拉近、负样本推远。这正是你副项目 Structure-CLIP / 难负样本的同一套数学，只不过那里是图文对，这里是文本对。

句向量的得到：编码器输出每个 token 的向量后，bge 取 `[CLS]` token 的向量（或做 mean pooling），再归一化（见第 2 节为什么要归一化）。

### 关键设计选择
- **为什么用 bge-small-zh 而非 OpenAI embedding**：①本地跑、免费、离线，不依赖你的代理是否提供 embedding 接口；②中文场景 bge 系列是 SOTA 级；③small 版 512 维，速度快、够用，后续可换 bge-m3（你主项目混合检索会用）。
- **维度 512 的权衡**：维度越高表达力越强但越慢、越占存储。small=512、base=768、large=1024。最小 demo 用 small。

### 面试高频问法
- "嵌入模型是怎么训练的？" → 对比学习 + InfoNCE，正负样本，拉近推远。
- "为什么语义相近的文本向量就相近？" → 因为训练目标就是这么设计的（对比损失逼出来的）。
- "InfoNCE 里温度 τ 的作用？" → 控制对相似度差异的敏感度，τ 小则分布尖锐。
- "句子级向量怎么从 token 向量得到？" → CLS / mean pooling + 归一化。

## 2. 向量检索与相似度（Vector Retrieval）

### 解决什么问题
有了文档向量库，给一个查询向量，怎么快速找出"最相关的 top-k 个文档"？答案：定义一个相似度度量，算查询和每个文档的相似度，取最高的 k 个。你代码里 `similarity_top_k=3` 就是取前 3。

### 核心原理（含数学）
**余弦相似度**（最常用）衡量两个向量方向的接近程度：

$$\text{cos}(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{\|\mathbf{a}\| \, \|\mathbf{b}\|} = \frac{\sum_{i=1}^d a_i b_i}{\sqrt{\sum_i a_i^2}\sqrt{\sum_i b_i^2}}$$

取值 $[-1, 1]$，越接近 1 越相似。它只看**方向**不看**长度**——这对文本检索很重要，因为文档长短不该影响语义相似判断。

**为什么常先归一化向量**：若把所有向量归一化成单位长度（$\|\mathbf{v}\|=1$），则：
$$\text{cos}(\mathbf{a}, \mathbf{b}) = \mathbf{a} \cdot \mathbf{b}$$
余弦相似度退化成点积，计算更快。而且此时点积、余弦、欧氏距离三者单调等价：
$$\|\mathbf{a}-\mathbf{b}\|^2 = \|\mathbf{a}\|^2 + \|\mathbf{b}\|^2 - 2\,\mathbf{a}\cdot\mathbf{b} = 2 - 2\,\mathbf{a}\cdot\mathbf{b}$$
（最后一步用了单位向量）。所以归一化后，"找余弦最大" = "找点积最大" = "找欧氏距离最小"。

你看到的检索结果 `相关度 0.450` 就是查询与该 chunk 的相似度分数。

**近似最近邻（ANN）**：文档量大时，逐个算相似度（暴力 KNN）太慢，复杂度 $O(N \cdot d)$。Chroma 等向量库用 HNSW（分层可导航小世界图）等索引，把检索降到近似 $O(\log N)$。代价是"近似"——可能漏掉极少数真正最近的点，换取百倍速度。

### 关键设计选择
- **top_k=3**：召回太少可能漏关键信息，太多会塞爆上下文、引入噪声、增加 token 成本。3 是 demo 合理起点，生产中常配 rerank 后再精筛。
- **余弦 vs 欧氏**：文本检索几乎都用余弦（或归一化点积），因为关注语义方向。

### 面试高频问法
- "向量检索为什么用余弦相似度？" → 只看方向不看模长，适合语义匹配。
- "归一化后余弦和点积、欧氏距离什么关系？" → 单调等价（推导见上）。
- "百万级文档怎么快速检索？" → ANN 索引（HNSW/IVF），用精度换速度。
- "HNSW 大概原理？" → 多层图，上层稀疏跳远、下层稠密精找，类似跳表。

## 3. 文本切块（Chunking）

### 解决什么问题
文档往往很长，不能整篇塞给模型：①嵌入模型有最大输入长度（bge 通常 512 token）；②检索粒度太粗（整篇）会让"相关的一句话"被大量无关内容稀释，相似度下降；③生成时上下文窗口和成本有限。所以要把文档切成合适大小的块（chunk）再分别嵌入。

你代码里 `SentenceSplitter(chunk_size=512, chunk_overlap=50)`。

### 核心原理（机制为主）
切块是检索质量的隐形决定因素，核心是两个超参的权衡：

**chunk_size（块大小）**：
- 太大 → 一个块含多个主题，嵌入向量是"语义平均"，检索精度下降，且可能超嵌入模型长度被截断。
- 太小 → 语义不完整（一句话脱离上下文），且块数暴增、检索开销大。

**chunk_overlap（重叠）**：相邻块共享一部分内容。作用是避免"关键信息正好被切在边界上"导致两个块都不完整。重叠 = 在边界处做冗余备份。

设文档长 $L$、块大小 $c$、重叠 $o$，则块数约：
$$N_{\text{chunks}} \approx \left\lceil \frac{L - o}{c - o} \right\rceil$$
（每个块净推进 $c-o$ 个单位）。overlap 越大，块数越多、冗余越多。

`SentenceSplitter` 比按字符硬切更聪明：它优先在句子边界切，尽量不切断完整句子，保证每个 chunk 语义自洽。

### 关键设计选择
- **512 + 50**：512 匹配 bge 的输入上限，50（约 10%）重叠是常见经验值，平衡冗余和完整性。
- **中文要注意**：512 这里指 token 近似，中文一个字约 1+ token，实际容纳的字数比英文少。
- **进阶方向**（主项目会用）：语义切块（按语义边界）、递归切块、按文档结构（标题层级）切——你主项目的 RAPTOR 就是层次化切块的代表。

### 面试高频问法
- "chunk_size 怎么定？太大太小各有什么问题？" → 见上权衡。
- "为什么要 overlap？" → 防关键信息被切在边界。
- "有哪些进阶切块策略？" → 语义切块、递归、按结构、层次化（RAPTOR）。
- "切块对检索质量影响大吗？" → 极大，是 RAG 最易被忽视的隐形调参点。

## 4. 检索增强生成（RAG 全链路）

### 解决什么问题
大模型有三个硬伤：①知识截止（训练后的新知识不知道）；②幻觉（不知道也硬编）；③私有知识缺失（你公司的文档它没见过）。RAG 用"先检索相关资料、再让模型基于资料回答"来同时缓解这三点。

### 核心原理
RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）。形式化：

给定查询 $q$，先用检索器 $R$ 从知识库 $\mathcal{D}$ 取 top-k 相关文档：
$$\mathcal{C} = R(q, \mathcal{D}, k) = \text{top-}k_{d \in \mathcal{D}} \; \text{sim}(f(q), f(d))$$

再把检索结果 $\mathcal{C}$ 拼进 prompt，交给生成器 $G$（LLM）：
$$\text{answer} = G(\text{prompt}(q, \mathcal{C}))$$

关键洞察：模型的回答**以检索内容为依据**（grounding），而非凭参数记忆瞎编。这就是为什么 RAG 能返回"引用来源"——答案可溯源。你代码里 `resp.source_nodes` 打印的就是这个依据。

**你的全链路对应代码**：
1. 切块 + 嵌入 → `build_index.py`（离线，建库）
2. 查询嵌入 + 向量检索 top-3 → `query.py` 的 `as_query_engine(similarity_top_k=3)`
3. 拼 prompt + LLM 生成 → `Settings.llm` (OpenAILike 调你的 mimo 端点)
4. 返回答案 + 引用 → `resp` 和 `resp.source_nodes`

### 关键设计选择
- **嵌入本地 / 生成走 API**：解耦。嵌入是高频、可本地化的；生成需要强模型，走端点。
- **为什么这是"朴素 RAG"**：单一向量检索、无查询改写、无重排序、无多跳。你主项目要逐步升级到 Advanced/Modular RAG 和 GraphRAG。

### 面试高频问法
- "RAG 解决了大模型什么问题？" → 知识截止、幻觉、私有知识。
- "RAG 为什么能减少幻觉？" → 答案 grounding 在检索内容上，可溯源。
- "朴素 RAG 有什么局限，怎么改进？" → 检索质量瓶颈 → 查询改写、混合检索、rerank、GraphRAG（这正是你主项目的路线）。
- "RAG vs 微调，什么时候用哪个？" → RAG 适合知识频繁更新/需溯源；微调适合改变模型风格/能力。可结合。
