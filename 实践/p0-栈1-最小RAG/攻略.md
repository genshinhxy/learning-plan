## 第 1 步｜建知识库（你来跑）

把 `docs/` 里的文档切块、向量化、存进向量库。

**你来做**：

```
! cd "D:/培养/实践/p0-栈1-最小RAG" && "D:/Anaconda/python.exe" build_index.py
```

**该看到**（忽略一堆 warning 和乱码，关键是最后）：
```
读到 1 个文档片段
Generating embeddings: 100%|...| 2/2
索引完成，向量库已持久化到 ...chroma_db
```

**这一步发生了什么**（对应数据流）：
1. 读 `docs/sample.md`
2. 用 `SentenceSplitter` 切成块（chunk_size=512）
3. 每块用本地嵌入模型 `bge-small-zh` 变成 512 维向量
4. 向量写入 Chroma 向量库，持久化到 `chroma_db/`

> 这步是"离线建库"。文档没变就不用重跑，向量库会一直留着。

## 第 2 步｜提第一个问题（你来跑）

```
! cd "D:/培养/实践/p0-栈1-最小RAG" && "D:/Anaconda/python.exe" query.py "什么是GraphRAG？"
```

**该看到**（三段）：
- `=== 问题 ===` 你的问题
- `=== 回答 ===` 模型基于文档生成的回答
- `=== 引用来源 ===` 答案依据的文档片段 + 相关度分数

**这一步发生了什么**：
1. 你的问题 → 嵌入成向量
2. 在向量库里找最相似的 top-3 块（向量检索）
3. 把这 3 块 + 你的问题拼成 prompt，发给你的 mimo-v2.5
4. 模型基于这些资料回答，并返回引用

> 关键点：模型不是凭记忆瞎答，而是**基于检索到的资料**回答——这就是 RAG 减少幻觉的原理。终端中文是乱码（Windows GBK），但内容是对的。

**思考**：top_k 太小可能漏信息，太大引入噪声、费 token。这是个要权衡的参数。

## 第 4 步｜自测（确认你真懂了）

不许翻上面，先自己答，再对照：

1. `build_index.py` 和 `query.py` 分别在 RAG 流程的哪一步？为什么分两个脚本？
2. 你问一个问题，从问题到答案，数据依次经过哪几个环节？
3. 为什么加了新文档后必须重跑 `build_index.py`？
4. RAG 为什么能减少大模型的幻觉？

<details>
<summary>点开看参考答案</summary>

1. `build_index` = 离线建库（切块+嵌入+入库）；`query` = 在线问答（检索+生成）。分开是因为建库慢且只需做一次，问答要反复做。
2. 问题 → 嵌入成向量 → 向量库检索 top-k 相似块 → 拼进 prompt → LLM 生成 → 返回答案+引用。
3. 因为新文档还没被嵌入入库，不重建索引检索不到它。
4. 因为答案 grounding 在检索到的真实资料上，而非模型参数里的模糊记忆，且可溯源（给引用）。
</details>

## 附：我跑通时的参考结果

第 2 步问"什么是GraphRAG？"，mimo-v2.5 的回答要点（你的结果应类似）：
> GraphRAG 在传统 RAG 基础上引入知识图谱，先从文档抽取实体和关系构成知识图谱，检索时不仅做向量相似匹配，还能沿图结构做多跳推理，适合跨多文档、多实体关联的复杂问题。微软 2024 年提出代表性方案。

引用来源：`[1] 相关度 0.450 | sample.md` `[2] 相关度 0.376 | sample.md`

（你的相关度分数应该一致，因为嵌入模型是确定性的；回答文字可能略有不同，因为 LLM 有随机性。）
